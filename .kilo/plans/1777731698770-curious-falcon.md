# CardiKeep Feature Enhancement Implementation Plan

## Overview
This plan outlines the integration of three major feature enhancements into the CardiKeep mobile application:
1. **UX Refinement: Notification Modal Reconfiguration** - Convert bottom-sheet notifications to top-aligned modals with smooth animations
2. **Data Management: Automated History System** - Implement local SQLite database with background worker for periodic data logging and circular buffer management
3. **State Management: Synchronization & Connectivity Guard** - Add global connectivity monitoring with UI overlays and middleware protection

## Current Architecture Analysis
- **Framework**: Flutter with Dart
- **State Management**: setState-based (no global state management library)
- **Navigation**: BottomNavigationBar with 2 tabs (Home, Settings) - will need to add History tab
- **Data Flow**: Real-time position fetching every 5 seconds via TrackerService
- **Notifications**: SnackBar-based alerts (currently for anti-loss alerts)
- **Dependencies**: Basic HTTP, geolocation, maps - will need to add sqflite, workmanager, connectivity_plus, provider for state management

## Feature 1: Notification Modal Reconfiguration

### Requirements
- Replace SnackBar alerts with custom top-aligned modal
- Implement downward slide animation from viewport top
- Optimize layout for thumb ergonomics and accessibility
- Maintain current alert triggers (anti-loss distance threshold)

### Implementation Steps
1. **Create Custom Modal Widget**
   - New file: `lib/widgets/top_notification_modal.dart`
   - Custom dialog with slide transition from top
   - Configurable content, duration, and actions
   - Accessibility-compliant sizing and positioning

2. **Update Alert Logic**
   - Modify `home_screen.dart` anti-loss alert trigger
   - Replace `ScaffoldMessenger.showSnackBar()` with custom modal
   - Ensure modal appears above all content (overlay)

3. **Animation & Layout Adjustments**
   - Use `SlideTransition` with `Offset(0.0, -1.0)` to `Offset.zero`
   - Position modal with `MediaQuery` safe areas consideration
   - Adjust modal height/width for optimal thumb reach

4. **Testing & Validation**
   - Verify modal appears on distance threshold breach
   - Test animation smoothness on different screen sizes
   - Accessibility testing with screen readers

## Feature 2: Automated History System

### Requirements
- SQLite local database for history logs
- Background worker triggering every 30 minutes
- Circular buffer enforcing 25-entry limit (delete oldest)
- New History tab in bottom navigation

### Implementation Steps
1. **Database Schema & Models**
   - Add `sqflite` dependency to pubspec.yaml
   - New file: `lib/models/history_entry.dart`
   - Database schema: id, timestamp, latitude, longitude, accuracy, battery_level, rssi
   - Database helper class: `lib/services/database_service.dart`

2. **Background Worker Setup**
   - Add `workmanager` dependency
   - Configure background task in `main.dart`
   - Task callback: fetch current position and insert into DB
   - Handle periodic execution every 30 minutes

3. **Circular Buffer Logic**
   - Implement row count checking in database service
   - Automatic deletion of oldest entry when count reaches 26
   - Efficient query for maintaining rolling window

4. **History Tab Implementation**
   - Add third tab to `MainLayout` bottom navigation
   - New screen: `lib/screens/history_screen.dart`
   - ListView with history entries (date/time, location preview)
   - Pull-to-refresh for manual updates

5. **Real-time UI Updates**
   - Integrate with existing position updates in home_screen
   - Update history tab when new entries are logged
   - Handle background task completion notifications

### Dependencies to Add
- `sqflite: ^2.3.0`
- `workmanager: ^0.5.2`
- `path_provider: ^2.1.5` (already present)

## Feature 3: Synchronization & Connectivity Guard

### Requirements
- Real-time network connectivity monitoring
- Global "Disconnected" state in application state
- UI overlay/disable elements when offline
- Non-breaking middleware layer protecting core functionality

### Implementation Steps
1. **State Management Introduction**
   - Add `provider` dependency for global state
   - New class: `lib/providers/connectivity_provider.dart`
   - Global state: connection status, offline overlay visibility

2. **Connectivity Monitoring**
   - Add `connectivity_plus` dependency
   - Implement listener in connectivity provider
   - Handle network changes (WiFi ↔ Mobile ↔ None)

3. **UI Overlay Implementation**
   - Create overlay widget: `lib/widgets/connectivity_overlay.dart`
   - Semi-transparent overlay with connection status message
   - Disable interactive elements when disconnected

4. **Middleware Integration**
   - Wrap app in `MultiProvider` in `main.dart`
   - Integrate connectivity provider with existing screens
   - Protect critical operations (position fetching, command sending)

5. **Non-Breaking Implementation**
   - Ensure core app functions without connectivity guard
   - Graceful degradation when offline
   - Clear separation between connectivity logic and app logic

### Dependencies to Add
- `provider: ^6.1.1`
- `connectivity_plus: ^5.0.2`

## Overall Integration Strategy

### Dependency Management
1. Update `pubspec.yaml` with all new dependencies
2. Run `flutter pub get` and resolve any conflicts
3. Update iOS/Android native configurations for background tasks

### Architecture Changes
1. **Navigation**: Expand from 2 to 3 tabs, add History
2. **State**: Introduce Provider for global connectivity state
3. **Storage**: Add SQLite for history persistence
4. **Background**: Implement WorkManager for periodic tasks

### Testing Strategy
1. **Unit Tests**: Database operations, background worker logic
2. **Integration Tests**: Connectivity monitoring, modal animations
3. **UI Tests**: History tab display, overlay behavior
4. **Performance**: Monitor memory usage with background tasks

### Risk Mitigation
- **Stability**: Implement features incrementally with feature flags
- **Performance**: Optimize database queries and background task frequency
- **Battery**: Monitor impact of background work and connectivity monitoring
- **Compatibility**: Test on various Android/iOS versions and network conditions

### Rollout Plan
1. **Phase 1**: Notification modal (low risk, isolated change)
2. **Phase 2**: History system (medium risk, new tab and DB)
3. **Phase 3**: Connectivity guard (higher risk, global state changes)
4. **Testing**: Full regression testing between phases
5. **Deployment**: Gradual rollout with monitoring

## Success Criteria
- Notification modal provides better UX than current SnackBar
- History system maintains 25 most recent entries automatically
- Connectivity guard prevents crashes and provides clear offline feedback
- All existing functionality remains stable and performant
- Code follows Flutter best practices and is maintainable
